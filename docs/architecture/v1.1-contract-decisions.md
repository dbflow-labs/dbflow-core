## Decision 1: Workflow Context Compatibility

Preserve `WorkflowContextInterface` unchanged.

Additive companion contracts and services normalize legacy flat variables into namespaced context without requiring hosts to implement a new mandatory interface.

Existing `getWorkflowVariables()` arrays remain valid and map into the `context` namespace (and optionally into `model` when hosts later adopt the field catalog / explicit mapping APIs).

## Decision 2: Context Namespaces

Normalized context always exposes:

- `model` — explicitly mapped workflowable data only
- `starter` — normalized starter identity metadata
- `actor` — current actor or system actor
- `context` — caller/business variables (legacy `getWorkflowVariables()` home)
- `workflow` — system metadata (key, version id, instance id)
- `task` — optional task/node metadata; empty before task exists

System namespaces `workflow` and `task` (and reserved identity namespaces when written by Core) cannot be overwritten by user-provided context bags.

## Decision 3: Snapshot and Live Data

- Default data source for v1.0-compatible definitions: **snapshot** semantics as currently implied by start/approve variable reads and instance metadata.
- Future definitions may declare `live` through an additive definition field.
- Live resolution requires an explicit provider and is auditable in later stages.
- Stage 1.1-A validates the field shape and rejects publication when the `live_context` capability is not registered.

## Decision 4: Field Catalog

Optional `FieldCatalogProvider` (or equivalent) may expose field definitions with:

- path, type, label, nullable, sensitive, allowed operators
- optional description, enum values, source namespace

Hosts are not required to implement it. Types and operators are closed sets defined by Core enums.

## Decision 5: Safe Path Resolution

`ContextPathResolver` resolves approved `namespace.rest.of.path` values from normalized arrays only.

- Distinguishes missing vs present-null
- Rejects invalid syntax, prohibited namespaces/methods, object/magic traversal, eval, callbacks, env/container access, and DB queries during traversal
- Sensitive fields are operation-aware (`condition`, `action_template`, `audit_display`, `ui_preview`)

## Decision 6: Definition Schema Versioning

- Missing `schema_version` ⇒ legacy v1.0 contract
- Explicit `"1.0"` remains valid
- Explicit `"1.1"` is recognized for additive fields
- Unsupported/malformed versions are rejected
- Published v1.0 snapshots are never rewritten by Core
- Authoritative validation remains `WorkflowDefinitionValidator` → `BlueprintValidator`

## Decision 7: Reassignment vs Delegation

- **Reassignment**: task-specific effective-assignee change via existing `ReassignTask`; preserves original responsibility; remains lock/transaction protected.
- **Delegation**: future time-bounded assignment rule with separate persistence/lifecycle; may later migrate pending tasks by calling `ReassignTask` per task; must not overload `reassigned` status. Deferred to Stage 1.1-B.

## Decision 8: Assignment Provenance

Frozen terminology (DTO/enum only in Stage 1.1-A):

- `direct`
- `reassignment`
- `delegation`
- `escalation`

No persistence columns or historical backfill in this stage.

## Decision 9: Legacy Action Semantics

Exact v1.0 behavior remains:

1. `ActionManager` resolves handler by action key
2. Success continues traversal
3. Failure continues unless `stop_on_error === true`
4. No persisted execution record, retry, queue, or exactly-once guarantee
5. Side effects may occur inside the workflow DB transaction

Normalized classification: `legacy_sync`.

## Decision 10: Reliable Action Compatibility

Future modes (`reliable_blocking`, `reliable_non_blocking`) are explicit additive definition values.

Absence of mode ⇒ `legacy_sync`. Never silently opt old definitions into queued/reliable execution. Reliable modes cannot publish until Stage 1.1-D registers the `reliable_action` capability and implements the ledger/runtime.

`stop_on_error` mapping under reliable modes is documented for later stages and not executed here.

## Decision 11: Runtime Capability Validation

Core-owned `RuntimeCapabilityRegistry` gates publication of unsupported semantics.

Stage 1.1-A enabled `context_schema_v1_1` only.

Stage 1.1-C additionally enables `sla` after the SLA runtime ships.

Stage 1.1-D additionally enables `reliable_action` and `outbound_webhook` after the reliable action ledger, queue runtime, and outbound webhook handler ship.

## Decision 13: Reassignment and Delegation Runtime (Stage 1.1-B)

- `ReassignTask` remains the only task-specific transfer path; optional idempotency key is additive.
- Delegation rules are time-bounded, computed-lifecycle, immutable after create (revoke + recreate to change).
- Resolution is direct-only with scope precedence node > workflow > global.
- Existing pending tasks are not auto-migrated; `MigratePendingTasksToDelegate` is explicit.

## Decision 12: Cross-package Responsibilities

- **Core**: contracts, validation, capability registry, normalization
- **Filament**: display/invoke Core; remain Pro-independent
- **Pro**: produce Core-compatible definitions; never become validator source of truth

## Rejected Alternatives

- Replacing `WorkflowContextInterface`
- Overloading `ReassignTask` as Delegation
- Silently converting all v1.0 Action nodes to queued reliable Actions
- Accepting definitions whose runtime semantics are not installed
- Making Pro the source of workflow validation truth

## Compatibility Matrix

| Concern | v1.0 | v1.1-A |
| --- | --- | --- |
| Context interface | `getWorkflowVariables()` | unchanged; additive normalizer |
| schema_version missing | treated as 1.0 | same, explicit version object |
| ReassignTask | available | unchanged behavior |
| Action missing mode | sync legacy | normalize to `legacy_sync` |
| Reliable/Delegation/SLA/Webhook publish | n/a | rejected via missing capability |
| Published snapshots | immutable by convention | still not rewritten |

## Stage Boundaries

Stage 1.1-A implements contracts, DTOs, validators, capability gates, docs, and tests only.

Deferred:

- 1.1-B Delegation runtime/persistence
- 1.1-C SLA runtime (reminders, overdue, basic escalation) — **implemented**
- 1.1-D Reliable Action ledger/queue + Webhook transport — **implemented**
- 1.1-E/F Filament/Pro UI
