## Core Models and Relationships

- `Workflow` → versions, currentVersion, activeVersion, instances; lifecycle helpers in model.
- `WorkflowVersion` → workflow, instances; `definition()` returns JSON snapshot array.
- `WorkflowInstance` → workflow, workflowVersion, workflowable, tasks, logs; `definition()`/`blueprint()` always from bound version.
- `WorkflowTask` → instance, assignments, logs.

## Workflow Definition Schema

Confirmed constants: `src/Definitions/WorkflowDefinitionSchema.php`.

Root fields: `key`, `name`, `version`/`schema_version`, `description`, `metadata`, `nodes`, `transitions`.

Node types: `start`, `approval`, `condition`, `action`, `end`.

Approval modes: `any`, `all`, `sequential`.

Assignee types: `user`, `role`, `permission`, `callback` (schema). Runtime-supported without host role integration: `user`, `permission`, `callback` (`runtimeSupportedAssigneeTypes()`). `role` fails open-core validation as unsupported.

Action config accepts `action_key` or legacy alias `action`; serialization prefers `action_key` (`src/Definitions/Nodes/ActionNode.php`).

Condition routing: runtime predicates are on `transition.condition`; node `config.expression` is documentary (`conditionPredicateField()` comment and `BlueprintValidator` checks).

UI metadata/`position` are opaque to the engine (`tests/Feature/Engine/MetadataImmunityTest.php`).

## Workflow Versioning and Immutability

Confirmed:

- Draft stored on `dbflow_workflows` draft columns (`SaveWorkflowDraft`, `CreateWorkflowDraft`).
- Publish snapshots draft JSON into a new `dbflow_workflow_versions` row (`PublishWorkflowDraft`).
- Instances bind `workflow_version_id` at start and continue using that snapshot (`StartWorkflow`, `WorkflowInstance::definition()`).
- Published immutability is an action-layer convention; schema still allows Eloquent updates.

## Task and Assignment Semantics

Confirmed:

- One task has many assignment rows (one per candidate user).
- Candidates are materialized as rows, not JSON arrays.
- Active work is identified by assignment `status = pending` (plus sequential lowest `sequence`).
- Actors are recorded via `acted_at` and audit logs; there is no separate “effective actor” column.
- Reassignment already exists: pending source assignment → `reassigned`, new pending assignment created with same `sequence` (`src/Actions/ReassignTask.php`). Provenance is log-payload only.

## Approval Mode Semantics

Confirmed in `ApproveTask` / `RejectTask` and `tests/Feature/Engine/AnyAllApprovalFlowTest.php`:

- **any**: first approve completes task; remaining pending assignments → `skipped`.
- **all**: each assignee must approve; task completes when no pending remain.
- **sequential**: only lowest pending sequence may act; completion when no pending remain.

## Rejection and Return Semantics

Confirmed runtime strategies (`src/Enums/RejectStrategy.php`):

- `starter` — return to first approval after start;
- `previous_node` — return to previous approval predecessor;
- `specific_node` — return to supplied approval node key;
- `end` — terminate instance as rejected.

Reject marks acting assignment `rejected`, other pending assignments `skipped`, task `rejected`, then either creates a rollback task with incremented `iteration` or ends the instance.

## Transactions, Locks, and Idempotency

Confirmed:

- Runtime mutations wrap `DB::transaction()` in Start/Approve/Reject/Cancel/Reassign/ProcessTaskTimeouts/Publish.
- Approve/Reject/Reassign use `lockForUpdate()` on task and instance (reassign also locks source assignment).
- Concurrent starts guarded by `active_key` unique constraint plus running-instance assertion.
- No optimistic lock version column on tasks/instances/versions.
- No Action execution idempotency key or outbox.

Inferred risks for v1.1:

- Action handlers and Laravel events run inside the DB transaction; retries can duplicate external side effects.
- Publish/draft save lack workflow row locks; concurrent publish can race on version numbers (unique constraint is the backstop).

## Action Node Runtime

Confirmed end-to-end path:

1. Traverser reaches Action node (`src/Services/WorkflowNodeTraverser.php`).
2. Resolves handler via `ActionManager` by `action_key`/`action`.
3. Executes synchronously inside the surrounding runtime transaction.
4. On success logs `action_executed` and continues.
5. On failure logs `action_failed`, dispatches `ActionFailed`; continues unless `stop_on_error=true` (then throws `ActionExecutionFailedException` and rolls back).

Core built-in handlers: `log`, `local_status_update` (`DBFlowServiceProvider`).

Confirmed absences: Action execution persistence table, queue dispatch, retry ledger, exactly-once fencing beyond transactional coupling (`tests/Feature/Engine/ActionFencingTest.php` documents current fencing expectations).

## Authoritative Validator

Confirmed authoritative entry point: `DbflowLabs\Core\Validation\WorkflowDefinitionValidator` → `BlueprintValidator`.

- Publish / sync: strict validation.
- Draft save: non-strict (`strict: false`) and may persist invalid drafts with recorded errors/warnings.
- Service container currently binds validator **without** assignee resolver registry (`DBFlowServiceProvider`), so missing permission/callback resolvers are not rejected at publish time unless a registry-aware validator is constructed.

## Known Contract Gaps

1. **Blocker (planning):** Action reliability — no execution ledger, retries, or outbox; side effects can duplicate under transaction retry.
2. **Blocker (planning):** Delegation does not exist; cannot reuse reassignment rows without losing original responsibility semantics.
3. **P0 risk:** Reassignment provenance is log-only; no assignment-level original/effective ownership columns.
4. **P0 risk:** SLA baseline only supports `reject_end`; no reminder/overdue/escalation tables or states.
5. **P1 risk:** Validator container binding omits assignee resolver registry, so publish can accept unresolved permission/callback keys.
6. **P1 risk:** Pro Action authoring (`mock_webhook`/`mock_event`) is not the same contract as Core handler keys / Pro-registered `premium_webhook`.
7. **P2 debt:** Unused `task_skipped` enum/log event; docs in Filament about cancel availability are partially stale.

## Proposed v1.1 Additive Migration Impact

**Non-executable planning only. No migrations were created in Stage 1.1-0.**

| Proposed additive structure | Current v1.0 limitation | Why reuse is unsafe | Null/default strategy | Stage |
| --- | --- | --- | --- | --- |
| `dbflow_workflow_delegations` table | no delegation entity | reassignment mutates assignment status and would erase original responsibility if overloaded | append-only rows; no backfill | 1.1-B |
| Assignment provenance columns (`original_assignee_user_id`, `reassigned_from_assignment_id`, optional actor source) | provenance only in logs | reconstructing history from logs is lossy and non-indexed | nullable; leave historical null | 1.1-B |
| Task SLA columns beyond `due_at` (e.g. `overdue_at`, `escalated_at`, reminder markers) or dedicated SLA event table | only `due_at` + timeout reject | overloading metadata JSON loses queryability and audit integrity | nullable; do not fabricate historical due/reminder data | 1.1-C |
| `dbflow_workflow_sla_events` (or equivalent) | no reminder/escalation ledger | task row mutation alone cannot prove reminder idempotency | append-only; no historical synthesis | 1.1-C |
| `dbflow_workflow_action_executions` | Action results only in logs | logs cascade-delete and lack attempt/idempotency keys | append-only; no fabricated historical executions | 1.1-D |
| Optional lock/version column on tasks or instances | pessimistic locks only | not strictly required if lockForUpdate remains sufficient; decide after concurrency contract | nullable/default 0 if introduced | 1.1-A/B decision |
| Indexes on new foreign keys and idempotency unique keys | current indexes cover v1.0 paths | unique attempt keys needed for SLA/Action exactly-once | create with table | matching feature stage |

Mandatory principles: additive, nullable, non-destructive, no fabricated historical data, no rewrite of published v1.0 definitions or running instance semantics.
