# DBFlow v1.1 Scope

## Version Theme

Enterprise Approval Controls & Reliable Actions

## Product Boundary

DBFlow remains a **model-first approval workflow engine** for Laravel and Filament.

DBFlow is not:

- a general-purpose BPM platform;
- a BPMN implementation;
- a generic low-code platform;
- a form database builder;
- an iPaaS or Zapier/n8n replacement;
- an AI agent orchestration platform.

Core remains the single source of truth for workflow semantics. Filament displays and invokes Core behavior. Pro creates Core-compatible definitions and provides premium management UX. Core must never depend on Filament, Pro, DBForm, DBErp, or a specific host application.

## P0 Scope

- Workflow Context Contract;
- reassignment (extend the existing v1.0 `DBFlow::reassign()` contract where provenance/concurrency gaps remain);
- delegation;
- fixed-duration SLA (extend beyond the existing v1.0 `timeout.due_in` + `reject_end` baseline);
- reminders;
- overdue state;
- basic escalation;
- reliable Action runtime;
- outbound Webhook;
- Core audit and recovery;
- Filament display;
- Pro configuration and management;
- compatibility and migration.

## P1 Scope

- SLA dashboard;
- manual Action retry UI;
- delegation filters;
- execution filters;
- field catalog picker;
- notification handler integrations.

## Preview Scope

- optional DBForm adapter;
- Context JSON Schema export;
- advanced escalation extension interfaces;
- custom Action templates.

## Explicit Non-goals

- BPMN;
- general BPM;
- low-code database builder;
- DBForm as a Core dependency;
- AI Router;
- AI Transformer;
- agent orchestration;
- connector marketplace;
- OAuth integration center;
- inbound Webhooks;
- WordPress;
- complex business calendars;
- automatic approval or rejection on timeout beyond the already-supported `reject_end` policy unless explicitly contracted;
- multi-level escalation chains;
- unrelated approval matrix redesign.

## Package Responsibilities

| Package | Responsibility |
| --- | --- |
| `dbflowlabs/core` | Workflow definition schema, validation, runtime transitions, task/assignment semantics, audit logs, Action handler contracts, migrations, public runtime API. |
| `dbflowlabs/filament` | Standard Filament surfaces that query and invoke Core. No independent workflow semantics. |
| `dbflowlabs/filament-pro` | Visual builder and premium management UX that produces and consumes Core-compatible definitions. |

## Dependency Direction

```text
Core <- Filament <- Pro
```

Forbidden reverse dependencies:

- Core must not reference Filament or Pro;
- Filament must not reference Pro production classes;
- Core must not reference DBErp, DBForm, or host application classes.

## Compatibility Commitments

- v1.0 definitions remain valid under the authoritative Core validator;
- running v1.0 instances preserve semantics for the published version they are bound to;
- all v1.1 definition fields are additive;
- Pro must not produce definitions unsupported by the installed Core version;
- Stage 1.1-0 compatibility fixtures under `tests/Fixtures/V10/` are permanent regression anchors.

## Migration Principles

- Additive changes only;
- new columns should be nullable unless a safe default is proven;
- do not rewrite historical tasks;
- do not generate historical due dates;
- do not infer historical delegation provenance;
- do not fabricate historical Action executions;
- do not alter published v1.0 workflow definitions;
- do not modify running instance semantics;
- avoid destructive rollback expectations.

## Release Blockers

Any of the following blocks a v1.1 release:

- duplicate workflow progression under concurrent approve/reassign/action paths;
- duplicate SLA reminder, overdue, or escalation side effects;
- secret leakage into logs, definitions, or webhook payloads;
- reverse package dependencies;
- unsafe or destructive migrations;
- Pro/Filament validator mismatch against the authoritative Core validator;
- Action double-execution without an agreed recovery contract.

## Scope Change Policy

Any new feature must be deferred to a later backlog unless it is required to satisfy an already-approved v1.1 contract or release blocker.

## Baseline Note (Evidence from Stage 1.1-0)

The following capabilities already exist in the v1.0 baseline and must not be reimplemented as greenfield work:

- Task reassignment: `src/Actions/ReassignTask.php`, `DBFlow::reassign()`, assignment status `reassigned`.
- Fixed due-date timeout with `reject_end`: approval `config.timeout`, `src/Actions/ProcessTaskTimeouts.php`, `dbflow:process-task-timeouts`.
- Synchronous Action nodes with `action_key`/`action`, `stop_on_error`, and built-in `log` / `local_status_update` handlers.

v1.1 work in these areas must be **additive** (provenance, reliability, reminders/escalation, outbound webhook) rather than replacing the existing contracts.
