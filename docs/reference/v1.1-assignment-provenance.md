# DBFlow v1.1 Assignment Provenance

## Purpose

Describe how Stage 1.1-B persists original responsibility and effective actor data without rewriting v1.0 history.

## Assignment Identity

| Column | Role |
| --- | --- |
| `assignee_user_id` | Stable uniqueness key with `workflow_task_id`. For newly materialized direct/delegated rows this is the original responsible actor. After `ReassignTask`, the new pending row uses the reassignment target as `assignee_user_id` (v1.0-compatible). |
| `original_assignee_user_id` | Original responsibility preserved across reassignment (nullable; falls back to `assignee_user_id`). |
| `effective_assignee_user_id` | Actor currently allowed to act (nullable; falls back to `assignee_user_id`). |
| `assignment_source` | `direct`, `reassignment`, `delegation`, or `escalation` (nullable → treat as `direct`). |
| `delegation_id` | Delegation rule used at materialization time (immutable thereafter). |
| `previous_assignment_id` | Superseded assignment for reassignment chains. |
| `reassignment_operation_key` | Optional idempotency key unique per task. |

## Fallbacks for v1.0 Rows

- Missing `assignment_source` → `direct`
- Missing effective actor → `assignee_user_id`
- Missing original actor → `assignee_user_id`
- No historical delegation or reassignment chain is fabricated

## Authorization

An actor may act on a pending assignment when they are the effective actor (`effective_assignee_user_id`, falling back to `assignee_user_id`), or when they are the original responsible actor on `assignee_user_id` for direct and delegated rows.

`original_assignee_user_id` is audit metadata on reassignment and escalation rows; it does not grant inbox visibility or approve/reject/reassign rights after transfer. UI listing uses the same actionable query scope.

## Multiple Responsibilities / One Delegate

One effective actor may represent multiple original assignment rows. Approval-mode vote counts remain based on assignment rows, not unique effective actors.
