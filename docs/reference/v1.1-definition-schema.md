# DBFlow v1.1 Definition Schema

## Schema Version Behavior

| Input | Normalization |
| --- | --- |
| missing / empty | legacy `1.0` |
| `"1.0"` | v1.0 |
| `"1.1"` | v1.1 additive contract |
| other / non-string | rejected |

`DefinitionSchemaVersion` is the explicit version object. Published snapshots are not rewritten.

## v1.0 Compatibility

- Definitions without `schema_version` remain valid.
- Existing node/transition shapes remain valid.
- Action nodes without `execution_mode` normalize to `legacy_sync` and omit the field on export.
- Stage 1.1-0 fixtures under `tests/Fixtures/V10/` remain permanent regression anchors.

## Context-related Fields

Optional top-level:

```json
{
  "context_policy": {
    "data_source": "snapshot"
  }
}
```

`data_source: live` requires the `live_context` capability (not enabled in Stage 1.1-A).

Presence of `context_policy` requires `context_schema_v1_1` (enabled in Stage 1.1-A).

## Approval Node Governance (Stage 1.1-B)

Additive approval-node config:

```json
{
  "type": "approval",
  "config": {
    "reassignment": { "enabled": true },
    "delegation": { "enabled": true }
  }
}
```

Compatibility defaults:

- missing `reassignment` → enabled (preserves v1.0 reassignment behavior)
- missing `delegation` → disabled
- `delegation.enabled: true` requires the `delegation` runtime capability
- root-level `delegation` / `delegations` definition fields remain unsupported (rules live in `dbflow_workflow_delegations`)

## Approval Node SLA (Stage 1.1-C)

Additive approval-node config (requires `sla` runtime capability):

```json
{
  "type": "approval",
  "config": {
    "sla": {
      "due_after": "PT24H",
      "reminders": [{ "before_due": "PT2H", "channel": "event" }],
      "overdue": {
        "notify": true,
        "channel": "event",
        "escalation": {
          "type": "reassign",
          "target": { "resolver": "permission", "value": "workflow_admin" }
        }
      }
    }
  }
}
```

Legacy `config.timeout` remains valid for v1.0 compatibility. Conflicting timeout + SLA definitions are rejected unless provably equivalent.

See `docs/architecture/sla-runtime.md`.

## Future Assignment Governance Fields

Root/node keys such as `webhook` still require their runtime capabilities. Root/node `sla` requires the `sla` capability (enabled in Stage 1.1-C).

Provenance terminology is frozen in code as `AssignmentSource`:

- `direct`
- `reassignment`
- `delegation`
- `escalation`

See also `docs/reference/v1.1-assignment-provenance.md`.

## Legacy Action Definition

```json
{
  "type": "action",
  "config": {
    "action_key": "log",
    "payload": {},
    "stop_on_error": false
  }
}
```

Semantics remain synchronous via `ActionManager`.

## Future Reliable Action Definition Contract

Additive fields (enabled when Stage 1.1-D registers `reliable_action`):

- `execution_mode`: `legacy_sync` | `reliable_blocking` | `reliable_non_blocking`
- `allow_manual_skip`
- `max_attempts`
- `retry` (`max_attempts`, `backoff_seconds`)
- `timeout_seconds`

Absence of `execution_mode` never opts a definition into reliable/queued execution.

## Runtime Capability Validation

Authoritative path remains:

`WorkflowDefinitionValidator` → `BlueprintValidator`

Capability checks are Core-owned through `RuntimeCapabilityRegistry`.

## Examples

Valid v1.0:

```json
{
  "key": "basic",
  "name": "Basic",
  "nodes": [],
  "transitions": []
}
```

Valid v1.1 snapshot context:

```json
{
  "schema_version": "1.1",
  "context_policy": { "data_source": "snapshot" }
}
```

## Invalid Examples

- `"schema_version": "2.0"`
- `"context_policy": { "data_source": "live" }` before `live_context` is enabled
- `"execution_mode": "reliable_blocking"` before `reliable_action` is enabled
- top-level `"delegation"` / `"sla"` / `"webhook"` before those capabilities exist

## Publication Rules

- strict validation on publish
- missing capability → structured error code `missing_runtime_capability`
- missing assignee resolver registration → `missing_assignee_resolver` when registry is wired

## Import / Export Rules

- Do not rewrite published v1.0 JSON
- Do not inject v1.1 fields into v1.0 exports unless explicitly requested
- Pro must pass through Core validation and must not invent reliable modes yet

## Pro Compatibility

Pro remains a producer of Core-compatible definitions. Core remains the validator source of truth. Stage 1.1-A does not add Pro UI controls.
